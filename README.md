# DS_YP_Boosting_UrbanStay_Cancellation
Прогнозирование отмен бронирований в сети отелей UrbanStay

Проект выполнен в рамках учебного курса Data Science.  
Задача проекта - построить модель машинного обучения, которая заранее определяет риск отмены бронирования и помогает отелю снизить потери от незаполненных номеров.

## Цель проекта

UrbanStay столкнулась с высокой долей отмен бронирований. Это снижает загрузку отелей и приводит к прямым финансовым потерям.

Необходимо было построить модель, которая:

- прогнозирует вероятность отмены бронирования;
- помогает заранее выявлять рискованные брони;
- снижает расчетную долю отмен;
- не допускает сильного падения загрузки;
- дает положительный финансовый эффект.

Целевые бизнес-требования:

- доля отмен после внедрения модели - не более 10%;
- загрузка отеля не должна снизиться более чем на 8%;
- относительный прирост IR - не менее 50%.

## Данные

В проекте использовались две таблицы:

### Бронирования

Таблица содержит параметры бронирования:

- дата бронирования;
- канал продаж;
- количество взрослых и детей;
- тип питания;
- тип номера;
- наличие парковки;
- стоимость бронирования;
- срок до заезда;
- длительность проживания;
- история отмен и незаездов клиента;
- статус бронирования.

Целевой признак - `booking_status`.

Положительный класс - отмена бронирования.

### Отзывы

Таблица содержит отзывы клиентов:

- идентификатор клиента;
- идентификатор бронирования;
- дата отзыва;
- оценка проживания;
- текст отзыва.

Отзывы использовались только как историческая информация. Для текущего бронирования подтягивался ближайший предыдущий отзыв этого же клиента, оставленный до даты текущего бронирования.

## Важные методологические решения

Особое внимание было уделено предотвращению утечек данных.

В ходе работы был выявлен риск использования признака, связанного с фактом наличия отзыва по текущему бронированию. Такой признак может косвенно показывать, что проживание состоялось, а значит становится утечкой целевой переменной.

Поэтому были приняты следующие решения:

- `customer_id` использовался только как технический ключ для объединения данных;
- `customer_id` не передавался в модель;
- `booking_id` не передавался в модель;
- `review_date` не передавался в модель;
- признак наличия отзыва не использовался;
- отзывы использовались только как предыдущий клиентский опыт, доступный до даты текущего бронирования.

## Подготовка признаков

В проекте были сформированы дополнительные признаки:

- общая длительность проживания;
- количество гостей;
- стоимость на ночь;
- стоимость на одного гостя;
- наличие детей;
- наличие специальных пожеланий;
- история отмен и незаездов;
- доля отмен в истории клиента;
- месяц бронирования;
- день недели бронирования;
- оценка предыдущего проживания;
- давность предыдущего отзыва;
- очищенный текст предыдущего отзыва.

Текст отзывов обрабатывался через `TfidfVectorizer` внутри пайплайна. Это исключает обучение словаря на калибровочной и тестовой выборках.

## Разделение данных

Данные были разделены последовательно по времени:

- train - 17 702 строки;
- calibration - 5 901 строка;
- test - 5 901 строка.

Перемешивание не использовалось, так как модель должна обучаться на прошлом и применяться к будущим бронированиям.

## Моделирование

Были проверены модели:

- LogisticRegression;
- RandomForest;
- CatBoost;
- LightGBM;
- XGBoost.

Ключевой метрикой выбора была бизнес-метрика `Incremental_IR`, так как для задачи важен не только факт правильного предсказания, но и экономический эффект.

Для лучших моделей был выполнен подбор гиперпараметров с помощью Optuna.

Итоговая модель:

- `XGBoost_tuned`;
- калибровка вероятностей методом `isotonic`;
- оптимальный бизнес-порог - `0.35`.

## Метрики итоговой модели

На тестовой выборке итоговая модель показала следующие результаты:

|---|---:|
| ROC-AUC | 0.977 |
| Precision | 0.823 |
| Recall | 0.870 |
| F1 | 0.846 |
| TN | 4 110 |
| FP | 282 |
| FN | 196 |

| Incremental_IR | 123 610 500 руб. |

## Экономический эффект

До внедрения модели:

- доля отмен - 25.57%;
- загрузка отеля - 74.43%;
- IR_before - 185 953 500 руб.

После внедрения модели:

- доля отмен - 3.32%;
- загрузка отеля - 69.65%;
- IR_after - 309 564 000 руб.;
- Incremental_IR - 123 610 500 руб.;
- относительный IR - 66.47%.

Расчет загрузки после внедрения выполнен по консервативной логике: гарантированно загруженными считаются только бронирования из группы `TN`.

## Важность признаков

Для интерпретации модели использовались:

- встроенная важность признаков XGBoost;
- SHAP-анализ.

Наиболее значимые факторы:

- давность предыдущего отзыва;
- текст предыдущего отзыва;
- оценка предыдущего проживания;
- прошлые незаезды клиента;
- срок до заезда;
- канал бронирования;
- история клиента;
- наличие детей;
- длительность проживания;
- история отмен.

Модель опирается на бизнес-интерпретируемые признаки, а не на технические идентификаторы.

## Вывод

Итоговая модель позволяет заранее выявлять бронирования с высоким риском отмены. На отложенной тестовой выборке модель снизила расчетную долю отмен с 25.57% до 3.32%, удержала снижение загрузки в допустимых пределах и дала прирост `Incremental_IR` 123 610 500 руб.

Модель можно рекомендовать к пилотному внедрению при условии регулярного мониторинга качества, бизнес-эффекта и переобучения на новых данных.

## Рекомендации для бизнеса

1. Использовать модель как систему раннего предупреждения.

Для бронирований с высоким риском отмены можно заранее запускать коммуникации с клиентом: напоминания, подтверждение планов, персональные предложения.

2. Разделить действия по уровню риска.

Для среднего риска достаточно мягкого напоминания. Для высокого риска можно использовать более активное подтверждение бронирования.

3. Учитывать историю отзывов.

Если у клиента был негативный или спорный прошлый опыт, отелю стоит заранее проверить пожелания клиента и уточнить ожидания.

4. Регулярно контролировать модель.

Необходимо отслеживать долю отмен, загрузку, `Incremental_IR`, `Precision` и `Recall` на новых данных.
